【技术难点与解决方案】
  - 难点1：如何证明"真人在场"且不可伪造
  - 解决方案：原型采用指纹模块进行本地比对，仅输出匹配 ID 和 score，不输出原始生物数据；再由
  ATECC608A 内部私钥进行 ECDSA 签名，形成"指纹通过 + 硬件私钥签名"的双重证明链。关键技术：
  // 生物特征证明：本地比对，数据不出芯片
  // 密码学证明：硬件签名，私钥不可读

  - 难点2：如何将一次具体高风险操作与一次物理授权严格绑定，避免授权被复用
  - 解决方案：由 HumanLink SDK 为每次操作生成 Challenge 与
  H_doc，设备仅对当前操作对应的摘要进行签名，形成"一次按压 = 一份断言 =
  一次授权"的机制，确保授权记录与具体操作一一对应。关键技术：  
  //Challenge生成：操作参数+设备DID+nonce原子绑定
  //签名输入包含完整操作上下文

  - 难点3：如何实现可扩展的软硬多端协调设计
  - 解决方案：采用分层协议栈架构，实现硬件抽象层与业务逻辑层解耦，通过标准化JSON-RPC通信协议和模块化SDK设计，支持多硬件平台、多编程语言、多部署模式的无缝集成。

【关键技术突破】 
  不可伪造性保证:
  - 生物特征数据本地处理，原始数据不出芯片
  - 硬件私钥不可提取，仅可调用签名功能
  - 传感器序列号+指纹匹配+业务上下文的三重绑定
  - 密码学签名提供数学层面的防伪造保证

  可扩展的协议栈设计:  
  - 分层架构支持硬件模块替换
  - 版本化协议确保向后兼容
  - 标准化JSON接口支持多语言SDK开发
  - W3C VC兼容的数据格式促进生态集成

  软硬协调的系统工程:  
  - 固件端实现硬件抽象和安全计算
  - SDK端提供高层API和数据模型
  - 统一的错误处理和状态管理机制

Q/A: 

1. 为什么使用设备绑定DID？——初代协议只解决设备合法性（注册或注销拥有授权权限的设备），至于设备由多少个人管理，是由硬件是否支持活体检测比对决定的。人负责掌控对设备的控制权以及责任本地归属上的变动。
2. Agent发起请求的时候如何知道自己绑定的哪个用户？—— HumanLink 不处理 Agent 身份认证，这是 IAM/Gateway 层的职责，`userId` 必须由上游可信地传入。关键在于 Agent 调用平台 API 时，必须携带凭证（session token / API key / OAuth token），这个凭证在颁发时就已经绑定了用户身份。
